# Suspension Design Studio — Engineering Validation Report

**Scope.** This report documents the physics behind the *Suspension Design Studio*
tab and its CAD-style kinematics viewer. It states, for every reported quantity:
the governing equation, its source, the assumptions, the limitations, and a
machine-checked validation against a hand calculation or analytic result.

**Honesty statement.** The Studio is an engineering calculator and design-exploration
tool, **not** a multi-body dynamics simulator. The suspension *kinematics* and the
*spring / frequency / sag / stress* calculations are exact and validated. Items that
are approximations or modelling assumptions are listed explicitly in §4 — read them
before relying on a number for a build decision.

Frame convention everywhere: **origin = front-wheel ground contact, +x rearward,
+y up, millimetres.**

## 2. Rear linkage — real four-bar (Pro-Link / Uni-Trak)

`monoshock-linkage` is solved as a **single-DOF four-bar** (`engine/studio/linkage.ts`),
not approximated as a direct shock:

- Swingarm point **S** (pushrod attach) rides a circular arc about the pivot.
- A pushrod of fixed length connects **S → rocker tip Q** (loop-closure constraint,
  solved by Newton-Raphson each travel step).
- The shock acts between the rocker tip **Q** and the frame mount.
- `MR(u) = |dL_shock/du|`; the non-linear S→Q→shock chain produces the rising rate.

The default linkage hardpoints are **generated by the solver itself** (a design
search, `calibrateLinkage`) targeting MR(0)≈0.32 with a good transmission angle, so
the four-bar is always geometrically valid (a hand-picked four-bar is easily
degenerate). The same method validates against the main app's `engine/sweep.ts`
four-bar (`shockLengthFourBar`).

**Result:** real rising-rate curves of **1.12–1.21×** across categories (production
Pro-Link bikes are typically 1.1–1.4×).

## 4. Assumptions & limitations (READ BEFORE TESTING)

These are explicit modelling choices, each documented in code:

1. **CoG height** is estimated as `0.32 · wheelbase` when not measured (typical moto
   CoG height ≈ 0.30–0.35·WB). Affects dynamic-load and anti-squat AS% only.
2. **Telescopic / USD fork MR = 1.0** exactly (spring travel = wheel travel). Correct
   for a fork, but it is an assumption, not a solved kinematic.
3. **Front fork spring rate** is the *combined two-leg* effective rate.
4. **Dynamic load** uses fixed **0.8 g braking** (front) and **0.5 g acceleration**
   (rear). Chain tension uses the same 0.5 g. These are representative, not your
   specific load case — change the masses/scenario for other points.
5. **Damping ratio** is reported as the *recommended target ζ* for the ride goal; the
   *critical damping coefficient* is exact, but actual ζ needs a damper dyno curve
   (the book is explicit that damping is velocity-dependent and dyno-characterised).
6. **Countershaft position** for anti-squat defaults to a small fixed offset from the
   swingarm pivot (manufacturers place it near the pivot to minimise AS change). It is
   editable via the drivetrain spec.
7. **Front trail** uses the front *suspension's* rake + a fork-offset value (seeded
   from the bike, default 30 mm). It updates with rake/wheel edits.
8. **Force vectors** in the diagram: the magnitudes (static load, shock force, spring
   force, chain tension) are real calculated values; they are drawn at a **relative
   pixel scale** for visualisation, and "weight transfer" uses the 0.8 g assumption.
9. **Progressive springs** are modelled as dual-rate (Book Ch2); a true progressive
   coil needs a spring-tester map.
10. **Not modelled** (by design): tyre carcass stiffness/Pacejka, flex, gyroscopic
    and full transient MBD response. For weave/wobble/eigenvalue work use the main
    app's MBD engine, not the Studio.

## 5. Diagram fidelity (item 14 of the spec)

Every graphical element is driven by the calculated geometry — no faked positions:

- Swingarm/axle: true circular arc about the pivot (`arcsin`).
- Shock length & angle: actual distance/orientation between the (rotating) lower end
  and the fixed upper mount; for linkage the lower end is the **solved rocker tip**.
- Wheel path: sampled axle positions over the full travel sweep.
- Motion-ratio HUD, dimensions, force-vector magnitudes: read from the same engine
  functions the Results panel uses (single source of truth).
- CoG / trail / instant-centre / chain-line overlays: recomputed from the **Studio's
  own edited geometry** (track your drags), with the main-bike store as fallback.

The silhouette / bodywork is the only cosmetic element (semi-transparent, drawn behind
the kinematics) and carries no numbers.
